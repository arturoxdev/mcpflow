# Effort is an optional Task tag, not a required dimension

A **Task** carries an optional `effort` enum (`low` | `high`) — orthogonal to `priority` and to Sprint scheduling. The column is nullable in the database and stays nullable forever; `NULL` is a first-class state meaning "sin clasificar". Validation is asymmetric: the internal `TaskForm` requires the user to pick `low` or `high` (both at create and at edit), while the Public Inbox and the REST API accept the field as optional and admit `null` on entry. Once classified, a Task cannot be returned to `NULL` from the UI — `NULL` exists only for legacy rows and for entries that arrived through external paths pending review. Rendering is conditional: the Effort glyph (a Lucide icon — yellow `Zap` for `low`, green `Snail` for `high`) appears in `Card`, `SprintTaskCard`, and `TaskPoolPanel` only when `effort != null`; unclassified Tasks show nothing.

## Considered Options

- **NOT NULL with a default value (`low` or `high`)**, mirroring `priority` (`NOT NULL DEFAULT 'medium'`): rejected. Forces a default for legacy rows that were never classified, collapsing two distinct states ("never thought about it" vs "explicitly chose low") into one. Pollutes the data and removes the ability to surface unclassified Tasks for review.
- **Required everywhere (TaskForm + Public Inbox + REST API)**: rejected. The Public Inbox is filled by external clients with no context about the user's effort scale; forcing a guess produces noise. Likewise, AI agents creating Tasks via REST do not always have grounds to estimate effort. Asymmetric validation keeps clean classifications inside the user's own workflow without polluting external entry points.
- **Allow editing back to `NULL`**: rejected. If a user opens an old `NULL` Task and the form lets them save it again as `NULL`, the unclassified pool never shrinks. Forcing a choice on edit guarantees that any Task touched after this change leaves the unclassified pool, which converges over time.
- **A separate `effort_classified_at` flag** to distinguish "explicit null" from "never set": rejected. Adds a column purely to encode metadata about another column. The asymmetric validation already gives us the same invariant: `NULL` only enters via legacy data or external creation paths, never from the internal UI.
- **Three values (`low` / `medium` / `high`)** to match `priority`'s shape: rejected. The user's mental model is binary — "lo despacho rápido" vs "necesito sentarme un rato". A middle bucket would absorb everything and erode the signal. Two values force the decision.

## Consequences

- **Two unrelated nullability rules now coexist on Task.** `priority` is `NOT NULL DEFAULT 'medium'`; `effort` is `NULL` with no default. Future readers will ask why — this ADR is the answer. The asymmetry reflects that priority is intrinsic to every Task (everything has *some* relative urgency) while effort is a tag the user opts into.
- **The unclassified pool is a real cohort.** A query `WHERE effort IS NULL` will return both legacy Tasks and Tasks created via Public Inbox / REST that haven't been triaged. This is intentional — it gives the user a backlog of Tasks to classify when convenient.
- **Edit form has stricter rules than create form's "no default".** Both create and edit require the user to pick `low` or `high` before saving. The edit form, when loading a Task with `effort = NULL`, presents the picker unselected and blocks save until classified. This is the only path that takes a Task out of the unclassified pool from the UI.
- **The Public Inbox `CreateTask` payload accepts `effort` as optional.** The REST API's `POST /api/boards/:boardId/tasks` does the same. Both default to `NULL` when omitted. AI agents that *do* have a grounded estimate can pass `low` or `high` and the Task enters classified.
- **Effort does not interact with Sprint planning at the DB level.** The Sprint planner displays the Effort glyph on `SprintTaskCard` for visual context, but no constraint, counter, or query treats `effort` specially when scheduling. A future feature could use it (e.g., a daily load indicator weighted by effort) without changing the schema.
- **Closed Tasks keep their `effort` value untouched.** Effort is independent of column state — a closed Task retains its classification for historical accuracy.
- **The Effort glyph is conditional.** Lists (`Card`, `SprintTaskCard`, `TaskPoolPanel`) render nothing for `NULL`-effort Tasks to avoid placeholder noise. The detail view and forms always show the picker.
